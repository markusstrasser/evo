╔══════════════════════════════════════════════════════════════════════════════╗
║                     THREE-OP KERNEL ARCHITECTURE                             ║
║              Tree Database with Transaction Pipeline                         ║
╚══════════════════════════════════════════════════════════════════════════════╝

┌─ Design Principles ────────────────────────────────────────────────────────┐
│                                                                            │
│ • Canonical State: A single, minimal source of truth for the tree          │
│   structure (:nodes, :children-by-parent).                                 │
│                                                                            │
│ • Derived Indexes: All other views of the data (parentage, siblings,       │
│   traversal order) are pure functions of the canonical state.              │
│                                                                            │
│ • Closed Operation Set: The system is manipulated by only three pure       │
│   operations (create, place, update). All complex behaviors are            │
│   composed from these primitives.                                          │
│                                                                            │
│ • Transactional Pipeline: Operations are processed in a strict,            │
│   multi-phase pipeline (normalize → validate → apply → derive) that        │
│   guarantees safety and consistency.                                       │
│                                                                            │
│ • Immutability: All operations are pure functions that take a database     │
│   value and return a new one. No in-place mutation occurs.                 │
│                                                                            │
│ • Extensibility: Higher-level features (e.g., structural editing) are      │
│   implemented as compilers that lower intents into core operations,       │
│   decoupling application logic from the kernel.                            │
│                                                                            │
└────────────────────────────────────────────────────────────────────────────┘

 1. Data Model (core.db)
═════════════════════════════════════════════════════════════════════════════

┌─ Canonical State ──────────────────────────────────────────────────────────┐
│                                                                            │
│ The canonical state is the minimal representation of the tree structure.   │
│                                                                            │
│  {:nodes              {Id → Node}       ; Node properties by ID            │
│   :children-by-parent {Parent → [Id]}   ; Ordered children for each parent │
│   :roots              #{:doc, :trash}   ; Root identifiers                 │
│   :derived            {...}}            ; All derived indexes (see below)  │
│                                                                            │
│  Example Data Shape:                                                       │
│    :nodes {"n1" {:type :p, :props {:text "A"}}}                             │
│    :children-by-parent {:doc ["n1"]}                                        │
│                                                                            │
└────────────────────────────────────────────────────────────────────────────┘

┌─ Derived Indexes (7 total) ────────────────────────────────────────────────┐
│                                                                            │
│ These indexes are computed by `derive-indexes` and provide efficient,      │
│ ready-to-use views of the tree for queries and higher-level logic.         │
│                                                                            │
│ 1. :parent-of {Id → Parent}                                                │
│    Purpose: O(1) lookup of a node's parent.                                │
│                                                                            │
│ 2. :index-of {Id → Int}                                                    │
│    Purpose: O(1) lookup of a node's position among its siblings.           │
│                                                                            │
│ 3. :prev-id-of {Id → Id | nil}                                             │
│    Purpose: O(1) lookup of the previous sibling.                           │
│                                                                            │
│ 4. :next-id-of {Id → Id | nil}                                             │
│    Purpose: O(1) lookup of the next sibling.                               │
│                                                                            │
│ 5. :pre {Id → Int}                                                         │
│    Purpose: O(1) lookup of a node's pre-order traversal index.             │
│                                                                            │
│ 6. :post {Id → Int}                                                        │
│    Purpose: O(1) lookup of a node's post-order traversal index.            │
│                                                                            │
│ 7. :id-by-pre {Int → Id}                                                   │
│    Purpose: O(1) lookup of a node ID by its pre-order index.               │
│                                                                            │
└────────────────────────────────────────────────────────────────────────────┘

┌─ Threading Patterns ───────────────────────────────────────────────────────┐
│                                                                            │
│ `derive-indexes` threads the canonical `db` through pure compute functions.│
│                                                                            │
│  db ─→ children-by-parent                                                  │
│      ├─→ compute-parent-of   → {:parent-of ...}                            │
│      ├─→ compute-index-of    → {:index-of ...}                             │
│      ├─→ compute-siblings    → {:prev-id-of ... :next-id-of ...}           │
│      └─→ compute-traversal   → {:pre ... :post ... :id-by-pre ...}         │
│                                                                            │
│ `compute-traversal` uses a nested recursive reduction:                     │
│   (letfn [(visit-node [state id]                                           │
│             (-> state                                                      │
│                 (update :pre-order conj id)   ; Pre-order visit            │
│                 (reduce visit-node children)  ; Recurse on children        │
│                 (update :post-order conj id)  ; Post-order visit           │
│             ))])                                                           │
│                                                                            │
└────────────────────────────────────────────────────────────────────────────┘

API Surface:
  (empty-db [] → Db)
    Returns a database with the correct canonical shape and empty collections.

  (derive-indexes [db] → Db)
    Recomputes all 7 derived indexes from the canonical state. Also runs any
    registered plugins and merges their derived data.

  (validate [db] → ValidateResult)
    Checks all database invariants. Returns {:ok? true} or {:ok? false,
    :errors [...]}. Invariants checked:
      • All children in :children-by-parent exist in :nodes.
      • No parent has duplicate children.
      • Each child has at most one parent.
      • All parents are either in :roots or :nodes.
      • No cycles in the parent chain.
      • No node is its own parent.
      • :derived indexes are fresh (match recomputation).

 2. Core Operations (core.ops)
═════════════════════════════════════════════════════════════════════════════

┌─ :create-node ─────────────────────────────────────────────────────────────┐
│                                                                            │
│ (create-node [db id type props] → Db)                                      │
│                                                                            │
│ Semantics:                                                                 │
│   Creates a new node shell in `:nodes`. The node is "unborn" and does not  │
│   appear in the tree structure until a `:place` operation is used.         │
│                                                                            │
│ Guarantees:                                                                │
│   • Idempotent: If a node with the given `id` already exists, the          │
│     operation is a no-op and returns the `db` unchanged.                   │
│                                                                            │
└────────────────────────────────────────────────────────────────────────────┘

┌─ :place ───────────────────────────────────────────────────────────────────┐
│                                                                            │
│ (place [db id under at] → Db)                                              │
│                                                                            │
│ Semantics:                                                                 │
│   Moves a node to a new parent at a specified position. This is the only   │
│   operation that modifies the tree structure.                              │
│                                                                            │
│ Algorithm (3 Phases):                                                      │
│   1. REMOVE: The node is unconditionally removed from its current parent,  │
│      if any. This ensures a node can only have one parent.                 │
│   2. RESOLVE: The `at` anchor is resolved to a concrete integer index      │
│      within the target parent's children list *after* the removal. This    │
│      is critical for stable reordering.                                    │
│   3. INSERT: The node is inserted into the target parent's children list   │
│      at the resolved index.                                                │
│                                                                            │
│ Anchor Resolution (5 types):                                               │
│   • :first      → index 0                                                  │
│   • :last       → index (count siblings)                                   │
│   • <integer>   → clamped to [0, count]                                    │
│   • {:before id} → index of `id`, or end if not found                      │
│   • {:after id}  → index of `id` + 1, or end if not found                   │
│                                                                            │
└────────────────────────────────────────────────────────────────────────────┘

┌─ :update-node ─────────────────────────────────────────────────────────────┐
│                                                                            │
│ (update-node [db id props] → Db)                                           │
│                                                                            │
│ Semantics:                                                                 │
│   Updates the properties of an existing node.                              │
│                                                                            │
│ Algorithm:                                                                 │
│   A deep (recursive) merge is performed on the node's `:props` map.        │
│   - Nested maps are merged recursively.                                    │
│   - Scalars and vectors in the new `props` overwrite old values.           │
│                                                                            │
│ Guarantees:                                                                │
│   • Safe: If the node `id` does not exist, it is a no-op.                   │
│                                                                            │
└────────────────────────────────────────────────────────────────────────────┘

 3. Transaction Pipeline (core.interpret)
═════════════════════════════════════════════════════════════════════════════

                      ┌────────────────────────┐
                      │      Input Ops         │
                      │      (Transaction)     │
                      └───────────┬────────────┘
                                  │
                   ╔══════════════╧══════════════╗
                   ║      1. NORMALIZE          ║
                   ╚══════════════╤══════════════╝
                                  │
              (normalize-ops [db ops] → ops')
         • Remove no-op :place (same parent/index)
         • Merge adjacent :update-node on same id
                                  │
                   ╔══════════════╧══════════════╗
                   ║      2. VALIDATE           ║
                   ╚══════════════╤══════════════╝
                                  │
        (validate-ops [db ops'] → [db' issues])
     Stateful reduction that applies ops as it validates.
     SHORT CIRCUITS on the first issue encountered.
                                  │
                   ╔══════════════╧══════════════╗
                   ║      3. APPLY (implicit)   ║
                   ╚══════════════╤══════════════╝
                                  │
     The final valid DB state from the validation
     phase is passed forward.
                                  │
                   ╔══════════════╧══════════════╗
                   ║      4. DERIVE             ║
                   ╚══════════════╤══════════════╝
                                  │
              (derive-indexes [db'] → final-db)
     Recomputes all 7 derived indexes for the final state.
                                  │
                                  ▼
                      ┌────────────────────────┐
                      │     InterpretResult    │
                      │ {:db, :issues, :trace} │
                      └────────────────────────┘

┌─ Validation Threading Pattern ─────────────────────────────────────────────┐
│                                                                            │
│ `validate-ops` uses a stateful reduction that threads the database state   │
│ forward, applying valid operations as it goes. This ensures that each      │
│ operation is validated against the state created by the previous one.      │
│                                                                            │
│  (reduce (fn [[db issues] op]                                              │
│            (if (seq (validate-op db op))                                   │
│              (reduced [db issues])      ; SHORT CIRCUIT on first error     │
│              [(apply-op db op) issues])) ; THREAD FORWARD db state         │
│          [initial-db []]                                                   │
│          ops)                                                              │
│                                                                            │
└────────────────────────────────────────────────────────────────────────────┘

API Surface:
  (interpret [db txs] → InterpretResult)
    Runs a full transaction through the 4-phase pipeline.

  (derive-db [db] → Db)
    Alias for `core.db/derive-indexes`.

  (validate [db] → ValidateResult)
    Alias for `core.db/validate`.

 4. Validation Semantics
═════════════════════════════════════════════════════════════════════════════

┌─ Issue Structure ──────────────────────────────────────────────────────────┐
│                                                                            │
│  {:issue :keyword     ; The type of error (e.g., :node-not-found)          │
│   :op     Op          ; The operation that failed validation               │
│   :at     Int         ; The index of the failing op in the transaction     │
│   :hint   String}     ; A human-readable explanation of the error          │
│                                                                            │
└────────────────────────────────────────────────────────────────────────────┘

┌─ Validation Rules by Operation ────────────────────────────────────────────┐
│                                                                            │
│ ┌─ :create-node ─────────────────────────────────────────────────────────┐ │
│ │ • :duplicate-create: Node with the same ID already exists.              │ │
│ └─────────────────────────────────────────────────────────────────────────┘ │
│ ┌─ :place ───────────────────────────────────────────────────────────────┐ │
│ │ • :node-not-found: The node being placed does not exist.                │ │
│ │ • :parent-not-found: The target parent is not a root or existing node.  │ │
│ │ • :anchor-not-sibling: A relative anchor (:before/:after) refers to an  │ │
│ │   ID that is not a child of the target parent.                          │ │
│ │ • :cycle-detected: The placement would make a node an ancestor of       │ │
│ │   itself. Cycle detection walks up the parent chain from the target     │ │
│ │   parent to see if it encounters the node being placed.                 │ │
│ └─────────────────────────────────────────────────────────────────────────┘ │
│ ┌─ :update-node ─────────────────────────────────────────────────────────┐ │
│ │ • :node-not-found: The node being updated does not exist.               │ │
│ └─────────────────────────────────────────────────────────────────────────┘ │
│ ┌─ All Ops ──────────────────────────────────────────────────────────────┐ │
│ │ • :invalid-schema: The operation map does not conform to the Malli      │ │
│ │   schema for its type.                                                  │ │
│ │ • :unknown-op: The `:op` key is not one of the three core operations.   │ │
│ └─────────────────────────────────────────────────────────────────────────┘ │
│                                                                            │
└────────────────────────────────────────────────────────────────────────────┘

 6. Structural Editing Layer (core.struct)
═════════════════════════════════════════════════════════════════════════════
This module is an EXTENSION EXAMPLE, demonstrating how to build higher-level
behaviors on top of the core operations.

┌─ Design Pattern: Intent Compilation ───────────────────────────────────────┐
│                                                                            │
│ 1. Define high-level "intents" as data (e.g., {:type :indent, :id "n1"}).  │
│ 2. Use a multimethod `compile-intent` to dispatch on the intent's `:type`. │
│ 3. Each method is a pure function `(fn [db intent] → [ops])` that reads    │
│    from the database's derived indexes to compute the necessary core ops.  │
│ 4. The results are concatenated into a single transaction vector.          │
│                                                                            │
│ Threading Pattern:                                                         │
│   intents ->> (mapcat #(compile-intent db %)) ->> vec                      │
│                                                                            │
│ This decouples application logic from the kernel. The kernel remains       │
│ small and stable, while features are added in separate modules.            │
│                                                                            │
└────────────────────────────────────────────────────────────────────────────┘

┌─ Example Intents ──────────────────────────────────────────────────────────┐
│                                                                            │
│ :delete {:id "id"}                                                         │
│   • Reads: Nothing.                                                        │
│   • Emits: [{:op :place, :id "id", :under :trash, :at :last}]               │
│   • Safety: Delete is non-destructive; it's an archive operation.          │
│                                                                            │
│ :indent {:id "id"}                                                         │
│   • Reads: `prev-id-of` to find the new parent.                            │
│   • Emits: [{:op :place, :id "id", :under <prev-sibling>, :at :last}]       │
│   • Safety: Emits `[]` (a no-op) if there is no previous sibling.           │
│                                                                            │
│ :outdent {:id "id"}                                                        │
│   • Reads: `parent-of` twice to find the grandparent.                      │
│   • Emits: [{:op :place, :id "id", :under <grandparent>, :at {:after <p>}}] │
│   • Safety: Emits `[]` if there is no grandparent.                         │
│                                                                            │
└────────────────────────────────────────────────────────────────────────────┘

 7. Usage Patterns
═════════════════════════════════════════════════════════════════════════════

┌─ Basic Workflow ───────────────────────────────────────────────────────────┐
│                                                                            │
│ (let [db0 (core.db/empty-db)                                               │
│       ops [{:op :create-node, :id "n1", :type :p, :props {}}                │
│            {:op :place, :id "n1", :under :doc, :at :first}]                 │
│                                                                            │
│       {:keys [db issues]} (core.interpret/interpret db0 ops)]              │
│                                                                            │
│   (if (seq issues)                                                         │
│     (println "Transaction failed:" issues)                                 │
│     (println "Success! New doc children:"                                  │
│              (get-in db [:children-by-parent :doc]))))                      │
│                                                                            │
└────────────────────────────────────────────────────────────────────────────┘

│     (if parent                                                             │
│       [{:op :place, :id id, :under parent, :at :first}]                     │
│       [])))                                                                │
│                                                                            │
│ ;; 2. Create
