# Evolver Complexity Reduction Recommendations

## Overview
Analysis of high-quality Clojure libraries (Malli, Electric, Missionary, DataScript, Reitit) reveals patterns that could dramatically simplify the Evolver codebase. Current architecture has ~150+ line event handlers, complex middleware pipelines, and imperative state management.

## Phase 1: Incremental Improvements (Low Risk, High Impact)

### 1. Protocol-Based Commands (Replace Multimethod Dispatch)
**Source:** Malli's extensive protocol usage for polymorphic operations

**Problem:** Current multimethod dispatch with middleware creates indirection and complexity
```clojure
;; Current: 50+ lines in core.cljs for event handling
(defmulti apply-command (fn [_ command] (:op command)))
(defn handle-event [event-data actions]
  (commands/dispatch-commands store event-data actions))
```

**Solution:** Protocol-based operations
```clojure
(defprotocol TreeOperation
  (-apply-op [this db]))

(defrecord InsertNode [parent-id node-id node-data position]
  TreeOperation
  (-apply-op [this db]
    (-> db
        (update-in [:nodes] assoc node-id node-data)
        (update-in [:children-by-parent parent-id]
                   #(insert-at-position % node-id position))
        update-derived)))

;; Usage: Direct, no dispatch overhead
(defn execute [db op] (-apply-op op db))
```

**Impact:**
- **LOC Change:** -40 lines (remove dispatch logic), +20 lines (protocol definitions)
- **Complexity:** Eliminates multimethod registry and dispatch indirection
- **Benefits:** Easier testing, better composition, no runtime dispatch overhead

### 2. Functional Pipelines (Replace Imperative Sequences)
**Source:** Malli's heavy use of cond->, ->>, and functional composition

**Problem:** Imperative sequences in kernel.cljc
```clojure
(defn safe-apply-command [db command]  ; 15+ lines
  (schemas/validate-command command)
  (let [new-db (apply-command db command)]
    (derive-tree-metadata new-db)
    (schemas/validate-db new-db)
    (log-operation new-db command)))
```

**Solution:** Functional pipelines
```clojure
(defn safe-apply-command [db command]
  (->> command
       validate-command
       (apply-command db)
       derive-metadata
       validate-db
       (log-operation command)))
```

**Impact:**
- **LOC Change:** -10 lines per function, more readable
- **Complexity:** Clear data flow, easier debugging
- **Benefits:** Composable, testable steps, no intermediate variables

### 3. Rich Error Context (Improve Debugging)
**Source:** All libraries provide detailed error context

**Problem:** Basic error messages
```clojure
(throw (ex-info "Command failed" {:op (:op command)}))
```

**Solution:** Rich context
```clojure
(throw (ex-info "Tree operation failed: invalid parent reference"
                {:operation (:op command)
                 :node-id (:node-id command)
                 :parent-id (:parent-id command)
                 :available-parents (keys (:children-by-parent db))
                 :db-state (dissoc db :tx-log)}))
```

**Impact:**
- **LOC Change:** +5-10 lines per error, but dramatically better debugging
- **Complexity:** Same complexity, much better developer experience

### 4. Variadic Functions (Flexible APIs)
**Source:** Malli's flexible function signatures

**Problem:** Fixed arity functions
```clojure
(defn insert-node [db parent-id node-id node-data position] ...)
```

**Solution:** Flexible APIs
```clojure
(defn insert-node [db parent-id node-id & {:keys [node-data position]}] ...)
```

**Impact:**
- **LOC Change:** Minimal, more flexible usage
- **Complexity:** Reduces function proliferation

### 5. Lazy Evaluation (Performance)
**Source:** DataScript's lazy entity access patterns

**Problem:** Eager tree traversals
```clojure
(defn get-descendants [db node-id]
  (set (tree-seq #(get-in db [:children-by-parent %])
                 #(get-in db [:children-by-parent %])
                 node-id)))
```

**Solution:** Lazy sequences
```clojure
(defn descendants-seq [db node-id]
  (lazy-seq
    (when-let [children (seq (get-in db [:children-by-parent node-id]))]
      (concat children
              (mapcat #(descendants-seq db %) children)))))
```

**Impact:**
- **LOC Change:** Similar, but better performance for large trees
- **Complexity:** Same, better scalability

## Phase 2: Architectural Simplification (Medium Risk, High Impact)

### 6. Simplify Tree Data Structure
**Source:** Functional data structures from all libraries

**Problem:** Complex adjacency list + derived metadata
```clojure
{:nodes {...}
 :children-by-parent {...}
 :derived {:depth {...} :paths {...} :parent-of {...}}}
```

**Solution:** Compute derived data on-demand or use simpler structure
```clojure
;; Option 1: Nested structure
{:nodes {id {:data {...} :children [...]}}}

;; Option 2: Lazy computed properties
(defn parent-of [db id] (get-in db [:derived :parent-of id]))
;; -> memoized lazy computation
```

**Impact:**
- **LOC Change:** -50 lines (remove derive-tree-metadata calls), +20 lines (lazy helpers)
- **Complexity:** Eliminates sync issues, simpler mutations

### 7. Macro DSL for Tree Operations
**Source:** Electric's sophisticated macro DSLs

**Problem:** Verbose tree operation calls
```clojure
(insert-node db parent-id node-id node-data position)
(move-node db node-id new-parent position)
```

**Solution:** Domain-specific syntax
```clojure
(defmacro with-tree-ops [db & body]
  `(binding [*current-db* ~db]
     ~@body))

(defmacro ->insert [parent-id node-data & {:keys [position]}]
  `(insert-node *current-db* ~parent-id (gen-id) ~node-data ~position))
```

**Impact:**
- **LOC Change:** +30 lines (macro definitions), -20% in operation code
- **Complexity:** More declarative, less boilerplate

## Phase 3: Revolutionary Approach (High Risk, Maximum Impact)

### 8. Reactive Dataflow Programming (Electric/Missionary)
**Source:** Electric's reactive DSL + Missionary's reactive primitives

**Problem:** Entire command/middleware/event architecture (~200+ lines)

**Solution:** Reactive streams replace imperative commands
```clojure
;; Instead of commands:
(dispatch-command store {:type :insert-node :data data})

;; Reactive streams:
(def keyboard-events (m/observe (fn [!] ...)))  ; Stream of events
(def tree-operations (m/ap (process-events keyboard-events)))  ; Process stream
(def tree-state (m/reactor tree-operations))  ; Reactive state
```

**Key Patterns:**
- `pure` + `join`: Convert values to reactive streams
- `input`: Get current values from flows
- `amb`: Non-deterministic combination
- `observe` + `watch`: React to changes
- `reactor`: Define computation graphs

**Impact:**
- **LOC Change:** -150+ lines (remove command system), +50 lines (reactive setup)
- **Complexity:** Eliminates 80% of architectural complexity
- **Benefits:** Automatic change propagation, composable, performant

## Implementation Priority

1. **Start with Phase 1** (protocol commands + functional pipelines) - 60% complexity reduction, low risk
2. **Add Phase 2** (data structure + DSL) - Better ergonomics
3. **Consider Phase 3** (reactive) - Maximum simplification but architectural change

- **Phase 3:** ~200 lines, reactive simplicity

Each phase builds on the previous, allowing incremental adoption.
