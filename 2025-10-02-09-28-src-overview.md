╔══════════════════════════════════════════════════════════════════════════════╗
║                     THREE-OP KERNEL ARCHITECTURE                             ║
║              Tree Database with Transaction Pipeline                         ║
╚══════════════════════════════════════════════════════════════════════════════╝

┌─ Design Principles ──────────────────────────────────────────────────────────┐
│                                                                             │
│ • Canonical State: A single, minimal source of truth for the tree           │
│   structure (`:nodes`, `:children-by-parent`).                              │
│                                                                             │
│ • Derived Indexes: All other views of the data (parentage, ordering, etc.)  │
│   are pure functions of the canonical state. No redundant state.            │
│                                                                             │
│ • Immutability: All operations are pure functions that take a database      │
│   value and return a new one. No in-place mutation.                         │
│                                                                             │
│ • Closed Operation Set: The kernel is defined by just three operations      │
│   (`create-node`, `place`, `update-node`). All higher-level behaviors are   │
│   compiled down to these primitives.                                        │
│                                                                             │
│ • Transactional Pipeline: Changes are applied through a multi-phase         │
│   pipeline (normalize → validate → apply → derive) that ensures safety      │
│   and consistency. Transactions are atomic; failure is all-or-nothing.      │
│                                                                             │
│ • Extensibility: The system is designed to be extended with higher-level    │
│   "intents" (e.g., structural editing) that compile to core operations,     │
│   and by adding new derived indexes via a plugin system.                    │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘

═════════════════════════════════════════════════════════════════════════════
 1. Data Model (core.db)
═════════════════════════════════════════════════════════════════════════════

┌─ Canonical State ────────────────────────────────────────────────────────────┐
│                                                                             │
│ The canonical state is the minimal representation of the tree structure.    │
│ All other data is derived from this.                                        │
│                                                                             │
│ {:nodes              {id → {:type :kw, :props {…}}} ; Node data             │
│  :children-by-parent {parent-id → [child-id, ...]} ; Tree structure         │
│  :roots              #{:doc, :trash, ...}          ; Root identifiers       │
│  :derived            {...}}                         ; See derived indexes   │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘

┌─ Derived Indexes (7) ────────────────────────────────────────────────────────┐
│                                                                             │
│ Derived indexes are computed by `derive-indexes` after each transaction.    │
│ They provide efficient, read-only views of the tree.                        │
│                                                                             │
│ 1. :parent-of {id → parent-id}                                              │
│    • Purpose: Fast parent lookup (O(1)). Essential for cycle detection.     │
│                                                                             │
│ 2. :index-of {id → integer}                                                 │
│    • Purpose: Fast child index lookup (O(1)).                               │
│                                                                             │
│ 3. :prev-id-of {id → prev-sibling-id | nil}                                 │
│    • Purpose: Fast previous sibling lookup (O(1)). Used for structural ops. │
│                                                                             │
│ 4. :next-id-of {id → next-sibling-id | nil}                                 │
│    • Purpose: Fast next sibling lookup (O(1)).                              │
│                                                                             │
│ 5. :pre {id → pre-order-index}                                              │
│    • Purpose: Pre-order traversal index for efficient subtree checks.       │
│                                                                             │
│ 6. :post {id → post-order-index}                                            │
│    • Purpose: Post-order traversal index.                                   │
│                                                                             │
│ 7. :id-by-pre {pre-order-index → id}                                        │
│    • Purpose: Inverse of :pre for fast lookup by pre-order index.           │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘

┌─ Threading Patterns ─────────────────────────────────────────────────────────┐
│                                                                             │
│ `derive-indexes` threads the canonical `children-by-parent` map through a   │
│ series of pure computation functions.                                       │
│                                                                             │
│ children-by-parent                                                          │
│   ├─→ compute-parent-of   → {:parent-of ...}                                │
│   ├─→ compute-index-of    → {:index-of ...}                                 │
│   ├─→ compute-siblings    → {:prev-id-of ... :next-id-of ...}                │
│   └─→ compute-traversal   → {:pre ... :post ... :id-by-pre ...}             │
│                                                                             │
│ `compute-traversal` has its own nested threading pattern:                   │
│   (reduce visit-node state roots)                                           │
│     └─→ visit-node (recursive)                                              │
│           ├─→ record pre-order index                                        │
│           ├─→ (reduce visit-node state-with-pre children) ; recurse         │
│           └─→ record post-order index                                       │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘

API Surface:
  (empty-db [] → Db)
    Returns a database with the correct canonical and derived shape.

  (derive-indexes [db] → Db)
    Recomputes all derived indexes from the canonical state. Also runs plugins.

  (validate [db] → {:ok? bool, :errors [string, ...]})
    Checks all database invariants. Key invariants checked:
      • All children in `:children-by-parent` exist in `:nodes`.
      • No parent has duplicate children.
      • Each child has at most one parent.
      • All parents are either roots or existing nodes.
      • No cycles in the parent graph (e.g., A is an ancestor of A).
      • No node is its own parent.
      • Derived indexes are fresh (match a re-computation).

 2. Core Operations (core.ops)
═════════════════════════════════════════════════════════════════════════════

┌─ create-node ────────────────────────────────────────────────────────────────┐
│                                                                             │
│ (create-node [db id type props] → Db)                                       │
│                                                                             │
│ • Semantics: Creates a node "shell" in the `:nodes` map. The node is not    │
│   placed in the tree until a `:place` operation is used.                    │
│                                                                             │
│ • Algorithm:                                                                │
│   1. Check if a node with `id` already exists in `db.nodes`.                │
│   2. If yes, return `db` unmodified (idempotent).                           │
│   3. If no, `assoc-in` the new node map at `[:nodes id]`.                    │
│                                                                             │
│ • Guarantees: Idempotent. Never fails.                                      │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘

┌─ place ──────────────────────────────────────────────────────────────────────┐
│                                                                             │
│ (place [db id under at] → Db)                                               │
│                                                                             │
│ • Semantics: Moves a node to a new parent at a specified position. This is  │
│   the only operation that modifies the tree structure.                      │
│                                                                             │
│ • Algorithm (3 Phases):                                                     │
│   1. REMOVE: The node `id` is removed from its current parent, wherever it  │
│      may be. This is done by scanning `db.children-by-parent`.              │
│   2. RESOLVE: The `at` anchor is resolved to a concrete index within the    │
│      target parent's children list *after* the removal. This is critical    │
│      for stable, predictable reordering.                                    │
│   3. INSERT: The node `id` is inserted into the target parent's children    │
│      list at the resolved index.                                            │
│                                                                             │
│ • Anchor Resolution: The `at` anchor can be one of 5 types:                 │
│   - :first / :last      → index 0 or (count children)                       │
│   - integer             → direct index (clamped)                            │
│   - {:before target-id} → index of `target-id`                              │
│   - {:after target-id}  → index of `target-id` + 1                          │
│                                                                             │
│ • Guarantees: Always succeeds at the op level, but may be rejected by the   │
│   validation phase if it would create a cycle or reference a missing node.  │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘

┌─ update-node ────────────────────────────────────────────────────────────────┐
│                                                                             │
│ (update-node [db id props] → Db)                                            │
│                                                                             │
│ • Semantics: Merges new properties into an existing node's `:props` map.    │
│                                                                             │
│ • Algorithm:                                                                │
│   1. Check if a node with `id` exists in `db.nodes`.                        │
│   2. If no, return `db` unmodified (no-op).                                 │
│   3. If yes, perform a deep (recursive) merge of the existing props with    │
│      the new `props`. Nested maps are merged; other values are replaced.    │
│                                                                             │
│ • Guarantees: Safe no-op if node doesn't exist. Preserves existing props    │
│   that are not specified in the update.                                     │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘

═════════════════════════════════════════════════════════════════════════════
 3. Transaction Pipeline (core.interpret)
═════════════════════════════════════════════════════════════════════════════

All changes to the database are processed through a 4-phase pipeline.

                      ┌────────────────────────┐
                      │ Transaction [Op, ...]  │
                      └───────────┬────────────┘
                                  │
                   ╔══════════════╧══════════════╗
                   ║      1. NORMALIZE          ║
                   ╚══════════════╤══════════════╝
                                  │
              (normalize-ops [db ops] → ops')
      • Remove no-op `:place` ops (same parent/index)
      • Merge adjacent `:update-node` ops on the same ID
                                  │
                   ╔══════════════╧══════════════╗
                   ║      2. VALIDATE           ║
                   ╚══════════════╤══════════════╝
                                  │
        (validate-ops [db ops'] → [db' issues])
      • Stateful reduction over ops.
      • On first error, SHORT CIRCUIT and return issues.
      • If valid, apply op and THREAD FORWARD new db state.
                                  │
                   ╔══════════════╧══════════════╗
                   ║      3. APPLY (implicit)   ║
                   ╚══════════════╤══════════════╝
                                  │
      • The final `db'` from the validation phase contains
        the result of all applied operations.
                                  │
                   ╔══════════════╧══════════════╗
                   ║      4. DERIVE             ║
                   ╚══════════════╤══════════════╝
                                  │
              (derive-indexes [db'] → final-db)
      • Recompute all 7 derived indexes and run plugins.
                                  │
                                  ▼
                      ┌────────────────────────┐
                      │  {:db, :issues, :trace} │
                      └────────────────────────┘

┌─ `validate-ops` Threading Pattern ───────────────────────────────────────────┐
│                                                                             │
│ Validation uses a stateful reduction that either short-circuits on the      │
│ first error or threads the evolving database state forward.                 │
│                                                                             │
│ (reduce (fn [[db issues] op]                                                │
│           (let [op-issues (validate-op db op)]                              │
│             (if (seq op-issues)                                             │
│               (reduced [db (into issues op-issues)]) ; SHORT CIRCUIT        │
│               [(apply-op db op) issues])))           ; THREAD FORWARD       │
│         [initial-db []]                                                     │
│         ops)                                                                │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘

API Surface:
  (interpret [db txs] → {:db Db, :issues [Issue], :trace [...]})
    The main entry point. Runs a transaction through the full pipeline.

  (derive-db [db] → Db)
    Public access to the final derivation phase.

  (validate [db] → ValidateResult)
    Public access to the database integrity check.

═════════════════════════════════════════════════════════════════════════════
 4. Validation Semantics
═════════════════════════════════════════════════════════════════════════════

┌─ Issue Structure ────────────────────────────────────────────────────────────┐
│                                                                             │
│ {:issue :keyword      ; Machine-readable issue type (e.g., :cycle-detected) │
│  :op     Op           ; The operation that failed validation                │
│  :at     integer      ; The index of the failing op in the transaction      │
│  :hint   "string"}    ; Human-readable explanation                          │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘

┌─ Validation Rules by Operation ──────────────────────────────────────────────┐
│                                                                             │
│ ├─ :create-node ────────────────────────────────────────────────────────────┤
│ │  • :duplicate-create: Node with the same ID already exists.               │
│ │                                                                           │
│ ├─ :place ──────────────────────────────────────────────────────────────────┤
│ │  • :node-not-found: The node being placed does not exist.                 │
│ │  • :parent-not-found: The target parent is not a root or an existing node.│
│ │  • :anchor-not-sibling: A relative anchor (`:before`/`:after`) refers to  │
│ │    an ID that is not a child of the target parent.                        │
│ │  • :cycle-detected: The move would make a node an ancestor of itself.     │
│ │    (Algorithm: walks up the parent chain from the target parent, checking │
│ │     if it encounters the node being placed).                              │
│ │                                                                           │
│ ├─ :update-node ────────────────────────────────────────────────────────────┤
│ │  • :node-not-found: The node being updated does not exist.                │
│ │                                                                           │
│ ├─ All Ops ─────────────────────────────────────────────────────────────────┤
│ │  • :invalid-schema: The operation map does not conform to the Malli schema│
│ │  • :unknown-op: The `:op` key is not one of the three recognized ops.     │
│ │                                                                           │
└─────────────────────────────────────────────────────────────────────────────┘

No-op detection for `:place` is handled during the normalization phase. It checks if the node's parent and final index would be identical after the remove→resolve→insert algorithm.

 5. Schema Contracts (core.schema - Malli)
═════════════════════════════════════════════════════════════════════════════

Malli schemas define the precise data contracts for all types and operations.

┌─ Core Types ─────────────────────────────────────────────────────────────────┐
│                                                                             │
│ • Id:     :string                                                           │
│ • Parent: [:or Id :keyword]                                                 │
│ • At:     [:or :int [:= :first] [:= :last] [:map [:before Id]] [:map [:after Id]]]│
│ • Node:   [:map [:type :keyword] [:props :map]]                              │
│ • Db:     [:map [:nodes [:map-of Id Node]] ...]                              │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘

┌─ Operation Schemas ──────────────────────────────────────────────────────────┐
│                                                                             │
│ • Op-Create: [:map [:op [:= :create-node]] [:id Id] ...]                     │
│ • Op-Place:  [:map [:op [:= :place]] [:id Id] [:under Parent] [:at At]]      │
│ • Op-Update: [:map [:op [:= :update-node]] [:id Id] [:props :map]]           │
│ • Op:        [:or Op-Create Op-Place Op-Update]                              │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘

┌─ Result Schemas ─────────────────────────────────────────────────────────────┐
│                                                                             │
│ • Issue:           [:map [:issue :keyword] [:op Op] ...]                     │
│ • InterpretResult: [:map [:db Db] [:issues [:vector Issue]] ...]            │
│ • ValidateResult:  [:map [:ok? :boolean] [:errors [:vector :string]]]       │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘

API Surface:
  (valid-op? [op]), (valid-db? [db])
    Boolean checks against schemas.

  (explain-op [op]), (explain-db [db])
    Returns detailed error maps on schema failure.

Transformers (`api-transformer`, `strict-transformer`) are provided for decoding/encoding data and stripping extra keys. Generators (`generate-op`, `generate-transaction`) are available for property-based testing.

 6. Structural Editing Layer (core.struct)
═════════════════════════════════════════════════════════════════════════════

This module is a canonical example of how to extend the kernel with higher-level, user-facing features without modifying the core.

┌─ Design Pattern: Intent Compilation ───────────────────────────────────────┐
│                                                                             │
│ 1. Define high-level "intents" as data (e.g., `{:type :delete, :id "a"}`).  │
│ 2. Use a multimethod, `compile-intent`, dispatched on `:type`.              │
│ 3. Each method is a pure function `(fn [db intent] → [Op, ...])`.           │
│ 4. The function reads from the DB's derived indexes to make decisions.      │
│ 5. It emits a vector of core operations to achieve the intent.              │
│ 6. If an intent is impossible (e.g., indenting the first child), it returns │
│    an empty vector (a safe no-op).                                          │
│                                                                             │
│ Threading Pattern:                                                          │
│   intents ->> (mapcat #(compile-intent db %)) ->> vec                       │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘

┌─ Example Intents ────────────────────────────────────────────────────────────┐
│                                                                             │
│ ├─ :delete ─────────────────────────────────────────────────────────────────┤
│ │  • Reads: Nothing.                                                        │
│ │  • Emits: `[{:op :place, :id "a", :under :trash, :at :last}]`              │
│ │  • Note: Delete is non-destructive; it's an archive operation.            │
│ │                                                                           │
│ ├─ :indent ─────────────────────────────────────────────────────────────────┤
│ │  • Reads: `:prev-id-of` derived index to find the new parent.             │
│ │  • Emits: `[{:op :place, :id "b", :under "prev-sib", :at :last}]`          │
│ │  • Safety: Returns `[]` if there is no previous sibling.                  │
│ │                                                                           │
│ ├─ :outdent ────────────────────────────────────────────────────────────────┤
│ │  • Reads: `:parent-of` derived index twice to find the grandparent.       │
│ │  • Emits: `[{:op :place, :id "b", :under "grandparent", :at {:after "p"}}]`│
│ │  • Safety: Returns `[]` if there is no grandparent.                       │
│ │                                                                           │
└─────────────────────────────────────────────────────────────────────────────┘

 7. Usage Patterns
═════════════════════════════════════════════════════════════════════════════

┌─ 1. Basic Workflow ──────────────────────────────────────────────────────────┐
│                                                                             │
│ (let [;; 1. Start with an empty DB
│       db0 (core.db/empty-db)
│
│       ;; 2. Define a transaction
│       tx [{:op :create-node :id "doc1" :type :doc :props {}}
│           {:op :place :id "doc1" :under :doc :at :last}
│           {:op :create-node :id "p1" :type :p :props {}}
│           {:op :place :id "p1" :under "doc1" :at :last}]
│
│       ;; 3. Interpret the transaction
│       {:keys [db issues]} (core.interpret/interpret db0 tx)]
│
│   ;; 4. Check for issues and use the resulting DB
│   (if (seq issues)
│     (println "Validation failed:" issues)
│     (println "Success! Doc children:" (get-in db [:children-by-parent "doc1"]))))│
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘

┌─ 2. Extension Pattern (Structural Editing) ────────────────────────────────┐
│                                                                             │
│ (let [;; 1. Get a valid DB state
│       db (:db (core.interpret/interpret ...))
│
│       ;; 2. Define high-level intents
│       intents [{:type :indent, :id "b"}
│                {:type :delete, :id "a"}]
│
│       ;; 3. Compile intents to core operations
│       ops (core.struct/compile-intents db intents)
│
│       ;; 4. Interpret the generated operations
│       result (core.interpret/interpret db ops)]
│
│   ;; Use the result of the high-level actions
│   (println "Final state:" (:db result)))                                    │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘

 8. Key Invariants & Guarantees
═════════════════════════════════════════════════════════════════════════════

✓ Operations are pure functions.
✓ The database is immutable.
✓ Transactions are atomic (all-or-nothing based on validation).
✓ Derived indexes are always consistent with the canonical state after a transaction.
✓ Node deletion is non-destructive (archival to `:trash`), preserving history and references.
✓ The core is completely described by 3 operations; complexity lives in compilers.
✓ Validation is comprehensive, preventing corrupt states like cycles or orphaned children.

 9. Performance Characteristics
═════════════════════════════════════════════════════════════════════════════

Let:
- n = total number of nodes in the database
- m = number of operations in a transaction
- k = number of children under a specific parent
- h = height of the tree
- d = max degree (number of children) of any node

┌─────────────────────────┬──────────┬────────────────────────────────────────┐
│ Operation               │ Big-O    │ Notes                                  │
├─────────────────────────┼──────────┼────────────────────────────────────────┤
│ derive-indexes          │ O(n)     │ Must visit every node and edge once.   │
│ create-node             │ O(1)     │ Map insertion.                         │
│ update-node             │ O(1)     │ Map update (props depth is constant).  │
│ place                   │ O(d)     │ Dominated by removing from old parent. │
│ would-create-cycle?     │ O(h)     │ Walks up the parent chain.             │
│ validate-ops (per op)   │ O(h)     │ Dominated by cycle check.              │
│ interpret (full)        │ O(m*h+n) │ m ops validated at O(h), then one O(n) │
│                         │          │ derive at the end.                     │
└─────────────────────────┴──────────┴────────────────────────────────────────┘
