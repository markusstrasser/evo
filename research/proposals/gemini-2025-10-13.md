### Core Insight (1 Paragraph)

The key to solving this is to stop thinking in terms of transient scripts and start modeling the entire review lifecycle as a set of interconnected, stateful, and addressable **resources**. Each stage (Idea, Proposal, Spec, ADR) becomes a persistent resource with a unique URI. The current scripts become stateless **tools** that act as pure functions, transforming one resource type into another (e.g., an `Idea` resource is transformed into a set of `Proposal` resources). This creates a transparent, auditable, and composable data flow where provenance is embedded directly into the resource URIs, making the entire process AI-native and REPL-friendly.

---

### 1. Server Architecture: Composition is Key

A single monolithic server would recreate the coupling problem. The best approach is a main server that **composes** several smaller, domain-specific servers. This aligns with the FastMCP `mount` pattern and ensures each stage is decoupled.

**Architectural Diagram:**

```
        +---------------------+
        |   ReviewMCP (main)  |
        +---------------------+
                  |
      +-----------+-----------+-----------+-----------+
      |           |           |           |           |
.mount("/ideas",   .mount("/proposals",   .mount("/specs",   .mount("/adrs",
      |                 |                 |               |
+---------------+ +-----------------+ +-------------+ +-----------+
|   IdeasMCP    | |  ProposalsMCP   | |   SpecsMCP  | |  ADRsMCP  |
+---------------+ +-----------------+ +-------------+ +-----------+
- create()      - generate()        - create()      - create()
- list()        - rank()            - refine()      - list()
- get()         - list()            - get()         - get()
                - get()             - add_test()
```

-   **ReviewMCP**: The main entry point. Mounts the other servers. Doesn't have its own tools.
-   **IdeasMCP**: Manages the lifecycle of raw ideas.
-   **ProposalsMCP**: Wraps the `bb architectural-proposals` and `bb best-repos-research` scripts. Handles generation and ranking.
-   **SpecsMCP**: A new, crucial stage for refining a chosen proposal into a concrete, testable specification. This is where AI agents can iterate.
-   **ADRsMCP**: The final stage for creating immutable decision records from finalized specs.

---

### 2. Data Flow & Provenance Tracking

Provenance is tracked via a hierarchical URI scheme. The URI of a resource contains the ID of its parent, creating an unbreakable chain.

**Data Flow:**

1.  An `Idea` is created, getting a URI: `ideas://idea-001`
2.  This URI is passed to the proposal generation tool, which creates 15 `Proposal` resources, each linked to the idea:
    -   `proposals://idea-001/gemini-q1`
    -   `proposals://idea-001/codex-q1`
    -   ...
3.  A user or agent chooses one proposal URI and passes it to the spec creation tool, creating a `Spec` resource:
    -   `specs://gemini-q1/spec-a8b3`
4.  The agent iterates on the spec using its URI. The content changes, but the URI remains stable.
5.  The final spec URI is passed to the ADR creation tool, creating a permanent `ADR` resource:
    -   `adrs://spec-a8b3/adr-2025-10-13-01`

This flow makes the entire history traceable from any artifact. Given `adrs://spec-a8b3/...`, you know it came from `specs://.../spec-a8b3`, which came from `proposals://.../gemini-q1`.

```python
# specs_mcp.py
specs_mcp = FastMCP("specs")

@specs_mcp.tool
async def create_from_proposal(proposal_uri: str, ctx: Context) -> str:
    """Creates an initial spec from a chosen proposal."""
    proposal_content = await ctx.read_resource(proposal_uri)
    
    # Use an LLM to convert high-level proposal to a structured spec
    prompt = f"Convert this proposal into a concrete spec with sections for API signatures, data structures, and test cases:\n\n{proposal_content}"
    spec_content = await ctx.sample(prompt)

    proposal_id = proposal_uri.split("/")[-1]
    spec_id = "spec-" + uuid.uuid4().hex[:4]
    spec_uri = f"specs://{proposal_id}/{spec_id}"
    
    await ctx.write_resource(spec_uri, spec_content.text)
    return spec_uri

@specs_mcp.tool
async def refine(spec_uri: str, instruction: str, ctx: Context) -> str:
    """Refines a spec based on a given instruction. (AI-native iteration)"""
    current_spec = await ctx.read_resource(spec_uri)
    prompt = f"Given this spec:\n\n{current_spec}\n\nApply this instruction: {instruction}"
    refined_spec = await ctx.sample(prompt)
    
    # Overwrite the existing resource, preserving the URI
    await ctx.write_resource(spec_uri, refined_spec.text)
    await ctx.info(f"Refined spec {spec_uri}")
    return spec_uri
```

### 4. Resource Structure (URI Scheme)

The URI scheme is the backbone of the system. Resources are exposed via the standard `@mcp.resource` decorator, likely backed by a file system or a simple database.

-   **Ideas**: `ideas://{idea_id}`
    -   Example: `ideas://rework-transaction-pipeline`
-   **Proposals**: `proposals://{idea_id}/{proposal_id}`
    -   Example: `proposals://rework-transaction-pipeline/gemini-question-1`
-   **Proposal Rankings**: `proposals://{idea_id}/_rankings/{ranker_id}`
    -   Example: `proposals://rework-transaction-pipeline/_rankings/codex`
-   **Specs**: `specs://{proposal_id}/{spec_id}`
    -   Example: `specs://gemini-question-1/spec-a8b3`
-   **ADRs**: `adrs://{spec_id}/{adr_id}`
    -   Example: `adrs://spec-a8b3/adr-2025-10-13-01`

### 5. Integration Points

-   **Existing Scripts**: The `proposals_generate` and `proposals_rank` tools act as wrappers around the existing `bb` scripts. They execute the shell commands and then persist the output files as MCP resources, bridging the old and new systems.
-   **`eval` package**: The `specs_refine` tool is the perfect place for an AI agent to work. The agent can read the spec, generate code based on it, run tests using the `eval` package, and if the tests fail, call `specs_refine` again with the error message as the `instruction` to self-correct. This creates a tight "spec-code-test" loop.
-   **Claude Code**: A developer (or a master agent) interacts with this system via a `fastmcp.Client`. The workflow becomes a series of REPL commands, which is perfectly aligned with the project's philosophy.

### 6. Concrete Example Walkthrough

**Scenario**: Reworking the transaction pipeline.

1.  **Create Idea**:
    ```python
    # In a REPL or script
    result = await client.call_tool("ideas_create", {"content": "Rework transaction pipeline to be more extensible."})
    idea_uri = result["uri"] # "ideas://rework-tx-pipeline"
    ```

2.  **Generate Proposals**:
    ```python
    prop_uris = await client.call_tool("proposals_generate", {"idea_uri": idea_uri})
    # Returns a list of 15 URIs like "proposals://rework-tx-pipeline/gemini-q1", ...
    ```

3.  **Review and Select**: The developer (or an agent) reads a few proposals.
    ```python
    print(await client.read_resource(prop_uris[0]))
    ```
    They decide `grok-q3` is the most promising.

4.  **Create Spec**:
    ```python
    spec_uri = await client.call_tool("specs_create_from_proposal", {
        "proposal_uri": "proposals://rework-tx-pipeline/grok-q3"
    })
    # Returns "specs://grok-q3/spec-b4c1"
    ```

5.  **AI-Native Refinement**: The initial spec is good but lacks examples.
    ```python
    await client.call_tool("specs_refine", {
        "spec_uri": spec_uri,
        "instruction": "Add concrete Clojure code examples for the main transaction function."
    })
    # The resource at "specs://grok-q3/spec-b4c1" is now updated.
    ```

6.  **Create ADR**: The spec is finalized and implemented. Time to record the decision.
    ```python
    adr_uri = await client.call_tool("adrs_create", {
        "spec_uri": spec_uri,
        "decision": "Adopted the event-sourcing pattern for the transaction pipeline as defined in the spec.",
        "rationale": "This approach provides the best debuggability and extensibility, at a minor cost to performance, which is an acceptable tradeoff."
    })
    # Returns "adrs://spec-b4c1/adr-2025-10-13-01"
    ```

### Key Benefits and Tradeoffs

-   **Benefits**:
    -   **Auditable Provenance**: The URI structure provides a complete, unbreakable history.
    -   **Composable & Decoupled**: Each stage is a separate server, preventing lock-in.
    -   **AI-Native**: The structured resource/tool model is ideal for agents to interact with. The `specs_refine` tool creates a natural loop for iteration.
    -   **REPL-Friendly**: The entire process can be driven from a Clojure or Python REPL, fitting the project's core philosophy.
    -   **Explicit Decisions**: The ADR stage makes decision-making a first-class, mandatory part of the process.

-   **Tradeoffs**:
    -   **Initial Overhead**: Setting up the four servers and the resource backend (e.g., file-based storage) is more work than a single script.
    -   **Resource Management**: A decision needs to be made on where to store the resource content (e.g., a structured directory in `.mcp/resources/`, a git repo, or a simple database). A file-based approach is likely sufficient and fits the "simplicity" constraint.
