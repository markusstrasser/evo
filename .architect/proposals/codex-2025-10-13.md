# FastMCP 2.x API Reference (Terse)

## Core Primitives

### Server
```python
from fastmcp import FastMCP, Context

mcp = FastMCP("ServerName")

# Composition
mcp.mount("/prefix", other_mcp_server)  # Live link
mcp.import_server(other_mcp_server)     # Static copy

# Proxy
mcp = FastMCP.as_proxy("http://remote/mcp")
```

### Tools (POST-like, side effects)
```python
@mcp.tool
async def process(param: str, ctx: Context) -> dict:
    """Docstring becomes description"""
    await ctx.info("Progress update")
    return {"result": "value"}
```

### Resources (GET-like, read-only)
```python
# Static
@mcp.resource("config://version")
def version(): return "1.0"

# Dynamic template with {placeholders}
@mcp.resource("users://{user_id}/profile")
def profile(user_id: int): return {...}

# RFC 6570 query params (NEW in 2.14)
@mcp.resource("search://{?q,limit}")
def search(q: str, limit: int = 10): return [...]
```

### Prompts (interaction templates)
```python
@mcp.prompt
def summarize(text: str) -> str:
    return f"Summarize: {text}"

# With Message objects
from fastmcp.prompts import PromptMessage

@mcp.prompt
def structured(data: dict) -> list[PromptMessage]:
    return [PromptMessage(role="user", content=...)]
```

### Context (injected automatically)
```python
@mcp.tool
async def tool(param: str, ctx: Context):
    # Logging
    await ctx.info("message")
    await ctx.error("error")

    # LLM sampling (ask client's LLM)
    response = await ctx.sample("prompt")

    # Resource access
    data = await ctx.read_resource("uri")

    # Progress reporting
    await ctx.report_progress(50, 100)

    # HTTP requests
    resp = await ctx.http_request("GET", "https://api.example.com")
```

Core insight: Treat each review stage as a capability-specific MCP “capsule” that speaks the same review record schema; orchestrate them through a thin coordinator server that only routes context and enforces provenance so stages stay swappable while the solo dev (or Claude Code) can drop in anywhere in the pipeline.

**Architecture Outline**
- `review-coordinator` (FastMCP server)  
  - Mounts four stage servers: `/ideas`, `/proposals`, `/specs`, `/adrs`
  - Owns cross-stage resources: `review://runs/{run_id}/ledger`, `review://runs/{run_id}/status`
  - Provides orchestration tools: `start_run`, `advance_stage`, `summarize_run`
- Stage servers (all FastMCP)  
  - `ideas-mcp`: capture/normalize idea notes  
  - `proposals-mcp`: wraps existing `bb architectural-proposals` flow + ranking  
  - `specs-mcp`: generates & iterates spec documents with guard-rail prompts  
  - `adrs-mcp`: records decisions, links to commits/tests  
- Shared prompt library (`review-prompts/`) provides templates consumed by tools across servers so output shape stays consistent.

```python
# review_coordinator.py
from fastmcp import FastMCP, Context
from review_shared import ReviewRun, Stage, load_run, save_run

mcp = FastMCP("review-coordinator")
mcp.mount("/ideas", ideas_server)
mcp.mount("/proposals", proposals_server)
mcp.mount("/specs", specs_server)
mcp.mount("/adrs", adrs_server)

@mcp.tool
async def start_run(title: str, idea_uri: str | None = None, ctx: Context) -> dict:
    run = ReviewRun.create(title, idea_uri)
    await ctx.info(f"Review run {run.id} created")
    save_run(run)
    return run.summary()

@mcp.tool
async def advance_stage(run_id: str, target_stage: Stage, ctx: Context) -> dict:
    run = load_run(run_id)
    result = await run.advance(target_stage, ctx)
    save_run(run)
    return result

@mcp.resource("review://runs/{run_id}/ledger")
def ledger(run_id: str) -> list[dict]:
    return load_run(run_id).ledger

@mcp.prompt
def stage_prompt(stage: Stage, payload: dict) -> list[PromptMessage]:
    return build_prompt(stage, payload)  # shared templates
```

**Resource Scheme & Provenance**
- Every run has immutable ledger entries: `{"stage": "proposals", "artifact_uri": "proposal://…", "inputs": [...], "outputs": [...]}`.
- URIs embed run + artifact IDs: `proposal://run-20250318-1/p-03`.
- Coordinator persists ledger to `research/review-ledger/{run_id}.json` (read/write inside workspace).  
- Resources are read-only snapshots; mutating actions always create new versions so provenance is append-only.

**Data Flow Example**
1. `start_run` → returns run id `run-20250318-1`.
2. `ideas` stage: `ideas.capture` ingests `.archived/ideas/*.md`, normalizes summary, stores `ideas://run-.../i-01`.
3. Coordinator `advance_stage(run, Stage.PROPOSALS)` → calls `proposals.generate`, which shells out to `bb architectural-proposals`, uploads resulting MDs as resources, and records ranking URIs.
4. Dev picks a proposal via Claude Code calling `proposals.curate` to apply filters (e.g., “favor pipeline instrumentation”). Output includes recommended `proposal_id`.
5. `specs.draft(run, proposal_id)` produces spec skeleton (types, invariants, tests) using structured prompt; spec stored at `spec://s-07` with version list.
6. If spec passes quick REPL validation (maybe via `ctx.sample` calling lint/test prompts), coordinator marks stage done.
7. After implementation, dev (or agent) runs `adrs.record(run, spec_id, decision_text)` linking commit SHA, tests, outcomes. ADR resource references spec + proposal URIs, closing loop.
8. `review://runs/run-.../ledger` now shows artifact lineage; `summarize_run` tool can produce report for README/ADR.

**Integration Points**
- Existing babashka scripts stay in place; `proposals.generate` invokes them via subprocess and wraps outputs as MCP resources.
- `bb best-repos-research` becomes optional enrichment: expose results under `resource://research/run-.../` and allow specs tool to read them via `ctx.read_resource`.
- Claude Code (or other IDE MCP client) lists tools/resources; user can trigger `advance_stage`, open spec resources, or call `iterate` with inline feedback.
- Evaluation hooks: after each stage, coordinator can optional call `ctx.sample` with “quality gate” prompts (e.g., ensure spec explains invariants, references DB schema).

**Key Benefits & Tradeoffs**
- Keeps stages independently deployable (swap `specs-mcp` prompt strategy without touching proposals); coordinator ensures consistent ledger.
- Provenance becomes first-class via immutable resources and ledger references, making ADRs auditable.
- Solo developer stays in control: each stage tool is manual-triggered; no hidden automation loops.
- Tradeoff: more servers/processes to run; mitigated by running them in-process via `FastMCP` mounting to avoid extra deployment overhead.
- Future extensibility: additional research servers can be mounted and referenced in ledger without changing core schema; ADRs can include manual notes or external docs via URIs.
